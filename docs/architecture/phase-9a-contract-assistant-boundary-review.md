# Phase 9A：合同助手边界评审

## 0. 阶段结论

Phase 9A 的结论是：HouseFolio 可以进入“合同风险提示助手”主线，但第一步必须严格限定为“边界评审”，不能直接进入功能实现。

本阶段只建立产品边界、法律表达边界、数据边界、AI 边界、工程边界和后续阶段拆分。Phase 9A 不新增页面、不新增 API route、不新增本地数据 key、不接 OCR、不上传合同照片、不接 PDF 多页扫描、不做法规库、不做律师复核、不输出维权结论，也不把 AI 包装成法律判断。

合同助手的第一版方向应是：

    用户主动粘贴合同文本
    系统按条款切分
    规则库命中常见风险
    AI 解释风险原因
    生成追问清单、建议改写方向和签约前 checklist
    用户可导出审读报告
    Settings 能说明并清除合同审读本地数据

但以上内容仍是后续 Phase 9B 之后的实现路线，不是 Phase 9A 的实现结果。

---

## 1. 背景与进入 Phase 9A 的理由

Phase 8D 已经完成 UI/UX redesign 总回归与阶段收口。HouseFolio 当前已经从早期偏工程 Demo / Dashboard 的视觉表达，调整为更接近中国大陆年轻租客可理解的生活化私人找房决策工具。

在这个基础上，Phase 9 可以进入签约前场景：当用户已经完成候选房源整理、通勤参考、辅助比较和看房记录后，下一步真实决策节点是租赁合同确认。

合同助手不是把 HouseFolio 改造成法律服务平台，而是把“签约前检查”纳入私人找房决策链路的最后一环：

    候选房源清单
    辅助比较
    看房记录
    准备签约
    合同风险提示

这条链路符合 HouseFolio 的产品定位：帮助租客管理自己主动收集的候选房源与签约前材料，并在不越界的前提下辅助用户识别常见风险点。

## 2. 产品定位边界

### 2.1 合同助手是什么

合同助手应定位为：

    签约前合同风险提示工具
    常见租房合同风险点辅助识别工具
    追问清单与签约前 checklist 生成工具
    条款理解与风险解释辅助工具

它服务的是普通租客在签约前的低成本检查场景：

    我是否看懂了这份合同？
    哪些条款需要进一步追问？
    哪些表述可能影响押金、违约金、维修责任或提前解约？
    我应该向房东或中介确认什么？
    哪些条款建议保留证据或要求写得更清楚？

### 2.2 合同助手不是什么

合同助手不能被定位为：

    AI 律师
    法律审查系统
    违法条款判定器
    霸王条款检测器
    自动维权工具
    合同有效性判断工具
    全国法规适配工具
    律师平替工具
    保证避坑工具

尤其不能宣传：

    判定违法
    判断无效
    保证无遗漏
    保证不踩坑
    帮你维权
    自动索赔
    比律师更便宜
    直接告诉你能不能签

正确的用户理解应是：

    HouseFolio 帮我在签约前看出一些常见风险信号，但最终是否签约、是否修改条款、是否寻求专业法律意见，仍由我自己决定。

## 3. 法律表达边界

### 3.1 允许使用的表达

Phase 9 后续产品文案可以使用：

    常见风险提示
    签约前检查建议
    需要追问
    建议确认
    建议写清楚
    可能影响押金退还
    可能影响提前解约成本
    可能影响维修责任划分
    可能影响费用承担
    建议保留沟通证据
    建议咨询专业人士

这些表达保持在“提示、提醒、解释、建议确认”的范围内。

### 3.2 禁止使用的表达

Phase 9 后续产品文案不得使用：

    AI 律师
    法律审查
    判定违法
    违法条款
    霸王条款自动识别
    合同无效
    保证不踩坑
    保证审查无遗漏
    自动维权
    帮你索赔
    必须修改
    不能签
    可以直接起诉
    法院一定支持

这些表达会把产品从“风险提示工具”推向“法律判断工具”，与个人作品和 HouseFolio 当前责任边界不匹配。

### 3.3 推荐免责声明

后续进入 UI 或报告导出时，建议使用类似表达：

    本功能仅用于辅助识别常见租房合同风险点，并生成签约前追问清单和检查建议，不构成正式法律意见，不替代律师、仲裁机构、法院或行政机关判断。合同文本由用户自行提供，识别结果可能存在遗漏或误判，用户应结合实际情况独立判断，必要时咨询专业人士。

Phase 9A 只记录该免责声明方向，不写入 UI。

## 4. 三层架构中的合同助手边界

合同助手不是独立的“法律 AI”，而是 HouseFolio 三层架构在签约前场景下的受限延伸。

### 4.1 L1：合同输入层

L1 在合同助手中负责：

    接收用户主动提供的合同文本
    后续支持条款切分
    后续支持敏感信息识别与脱敏
    后续可评估合同照片 OCR

Phase 9 第一版只允许文本粘贴审读。合同照片、OCR、PDF 多页扫描全部后置。

L1 不得做：

    自动读取本机文件夹
    自动扫描相册
    自动上传合同照片
    自动读取 PDF
    自动从聊天记录中抽取合同
    自动抓取第三方合同页面
    未经用户确认直接把合同全文发给 AI

### 4.2 L2：风险规则层

L2 在合同助手中负责：

    条款规则匹配
    常见风险类型命中
    风险等级或关注等级给出
    结构化风险项输出
    后续可回归测试的规则维护

L2 可覆盖的第一批风险类型建议包括：

    押金退还
    违约金
    提前解约
    房东入室权
    维修责任
    费用归属
    转租限制
    单方解除
    房屋权属或出租资格
    续租与涨租
    物品清单
    交付状态
    付款方式与收据
    口头承诺未写入合同

L2 必须保持规则可解释。风险等级应来自规则库，而不是由 LLM 临场决定。

L2 不得做：

    法律结论
    条款违法判定
    合同整体有效性判断
    诉讼胜率判断
    维权路径结论
    自动适配全国地方政策

### 4.3 L3：AI 解释层

L3 在合同助手中只负责：

    把规则命中的风险翻译成人话
    解释为什么需要注意
    生成建议追问问题
    生成建议改写方向
    生成签约前 checklist
    组织审读报告文本

L3 不得做：

    风险等级判定
    法律效力判断
    违法判断
    排序替用户决定
    承诺无遗漏
    维权结论
    引用不存在或未经核验的法规
    假装已做律师复核

L3 的输入必须来自 L1 / L2 的结构化结果。不得让 LLM 直接读取原始合同后自由发挥成法律意见。

## 5. 数据与隐私边界

合同文本属于高敏材料。它可能包含：

    用户姓名
    身份证号
    手机号
    住址
    门牌号
    房东姓名
    房东手机号
    银行账户
    微信号
    签字页
    付款信息
    房产证信息
    证件照片
    房屋精确地址
    其他合同相对方个人信息

因此 Phase 9 必须沿用 HouseFolio 本地优先原则。

### 5.1 默认数据策略

后续实现时应默认：

    合同文本优先保存在本地
    AI 调用前必须用户确认
    发送给 AI 前应尽量脱敏
    服务端不长期保存合同原文
    服务端日志不记录合同原文
    不记录身份证号、手机号、银行卡号、门牌号、签名等敏感字段
    Settings 必须说明合同审读数据属于本地数据权利范围

### 5.2 AI 调用策略

后续接入 AI 时必须满足：

    用户主动点击触发
    触发前显示确认弹窗
    弹窗说明将发送哪些类型的信息
    尽量发送脱敏后的条款片段与规则命中结果
    不默认发送完整合同
    不发送身份证、银行卡、签字页、证件页
    不在服务端日志输出 prompt 原文
    不把 AI 输出包装为法律意见

### 5.3 本地数据权利

后续如果新增合同审读记录，必须同步考虑：

    查看本地合同审读数据
    导出合同审读报告
    删除单次审读记录
    清除全部合同审读记录
    Settings 中说明合同文本和审读结果是否保存
    src/lib/privacy/local-data.ts 纳入新增本地数据类型

Phase 9A 不新增 localStorage key。是否新增例如 housefolio:contract-reviews 等 key，应留到 Phase 9B 或后续数据模型评审。

## 6. OCR 与合同照片边界

Phase 9A 明确：当前阶段不做 OCR，也不做合同照片上传。

OCR 应属于 Phase 10，而不是 Phase 9A 或 Phase 9 MVP 起点。

原因：

    OCR 识别可能漏字、错字或断章
    合同照片可能包含签名、身份证、房产证、手机号等敏感信息
    多页 PDF / 多图排序 / 重复页 / 缺页检查都会增加工程复杂度
    “照片 → OCR → AI 自动审查”容易让用户误以为结果可靠
    第一版需要先验证文本版合同审读价值

后续如果进入 OCR，必须遵守：

    OCR 后必须让用户校对文本
    不允许“合同照片 → AI 自动判定”
    OCR 只是输入方式，不是审读内核
    审读内核应先在 Phase 9 文本版中完成

Phase 9A 不创建 OCR provider，不选型腾讯云或百度云，不上传图片，不做图片预览，不做 PDF 解析。

    lib/ai

    lib/local-store

    数据库 SDK
    浏览器文件系统底层存储
    第三方平台页面读取逻辑

Phase 9A 不决定最终文件结构，只记录边界：业务页面不得直接绑定平台能力，合同助手也不得突破 HouseFolio 既有封装原则。

## 12. 当前阶段结尾判断

Phase 9A 是必要的刹车阶段。

合同助手确实能增强 HouseFolio 的产品完整性，因为它连接了“看房决策”与“签约前检查”之间的真实用户路径。但它也天然更接近法律、隐私与责任边界，因此必须先建立边界，再做页面、规则、AI 与数据权利。

正确推进顺序是：

    先边界评审
    再路由与数据计划
    再文本版最小实现
    再规则库
    再 AI 解释
    再 OCR

不能反过来从 OCR、照片上传或 AI 法律判断开始。
