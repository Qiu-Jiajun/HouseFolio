# Phase 3H-1：看房照片/视频本机保存边界评审

## 1. 阶段定位

本阶段是 Phase 3H 之后插入的补丁评审阶段。

Phase 3H 已经完成 v2.0 本地优先数据权利层总收口。当前不应继续扩张照片能力，也不应直接实现视频存储。由于真实找房场景中，用户可能保存自己拍摄的视频，或者收到房东、中介发送的视频，因此需要先把“视频是否进入 HouseFolio”这个问题正式记录为边界评审，而不是直接改代码。

本阶段只写评审文档，不修改代码、不修改 UI、不修改 IndexedDB provider、不新增类型、不新增路由、不新增 localStorage key。

阶段名称：

Phase 3H-1：看房照片/视频本机保存边界评审

英文名称：

Phase 3H-1: Local Viewing Media Boundary Review

## 2. 当前结论

当前结论是：

短期继续保持“看房照片｜本机保存”，不要马上改成“看房照片/视频｜本机保存”。

中期可以将能力抽象为“看房媒体｜本机保存”或“看房照片/视频｜本机保存”，但必须先完成边界设计、存储策略、性能策略、隐私策略和数据权利策略。

长期可以把当前 ListingPhoto 抽象升级为 ListingMedia，但不应现在重构。当前照片功能刚稳定，贸然重构会破坏 Phase 3D–3H 已经形成的本地照片持久化、Settings 数据权利、JSON 导入和 Portfolio 首图闭环。

## 3. 视频是否属于 HouseFolio v2.0 本地优先资料

结论：属于，但不是当前立即实现的 P0。

理由：

- 视频和照片一样，都是用户看房现场资料；
- 视频可能比照片更能表达空间动线、户型走向、采光和实际居住感；
- 用户自己拍的视频符合“用户主动添加、默认私有、本地优先”的产品定位；
- 房东或中介发来的视频，也可能成为用户的私人决策资料；
- 但视频文件更大、隐私风险更高、UI 和存储复杂度更高，因此不能按图片逻辑直接扩展。

因此，视频可以进入长期路线图，但当前只进入“边界评审”，不进入实现阶段。

## 4. 为什么不能直接按图片逻辑加视频

视频相比图片多出明显复杂度：

| 维度 | 图片 | 视频 |
| --- | --- | --- |
| 文件大小 | 通常几百 KB 到几 MB | 很容易几十 MB 到几百 MB |
| 本机存储压力 | 可控 | 明显更高 |
| 加载性能 | 缩略图可控 | 列表页不应直接加载视频 Blob |
| 隐私风险 | 门牌、人脸、合同、室内细节 | 还可能包含声音、对话、定位线索、房东或中介声音 |
| 第三方内容风险 | 平台图片不能搬运 | 中介或房东视频也不能公开再传播 |
| AI 风险 | 不应默认进入 AI | 更不能默认视频理解、语音转写、OCR 或截图分析 |
| UI 复杂度 | 图库即可 | 需要播放器、封面、大小限制、加载策略 |

所以视频不是“照片模块支持 mp4 后缀”。它应被看作“本地看房媒体资料”的独立边界问题。

## 5. 照片和视频是否应合并为一个模块

长期建议合并，短期不改 UI。

最终更合理的用户可见模块名称是：

看房照片/视频｜本机保存

或者更抽象地称为：

看房媒体｜本机保存

但当前不应提前改 UI 标题。原因是：

- 当前系统尚不支持视频；
- 如果 UI 改成“照片/视频”，用户会误以为视频已经可用；
- 当前 Phase 3D–3H 的稳定能力是“看房照片｜本机保存”；
- 现阶段应避免为了未来能力而改变已稳定的用户预期。

因此，短期维持：

看房照片｜本机保存

未来真正实现视频后，再统一调整为：

看房照片/视频｜本机保存

## 6. 当前 ListingPhoto 是否需要升级为 ListingMedia

短期：不升级。

长期：可以评估升级。

当前模型是 ListingPhoto，已经服务于：

- Detail 本机照片展示；
- IndexedDB 照片 Blob 保存；
- Settings 照片数量与占用空间统计；
- 清除全部本机照片；
- Portfolio 首图读取；
- v2.0 本地优先数据权利闭环。

因此不应现在为了视频重构 ListingPhoto。

未来如果进入视频实现阶段，可以设计 ListingMedia：

- ListingMediaKind = photo | video
- id
- listingId
- kind
- fileName
- mimeType
- sizeBytes
- createdAt
- updatedAt
- storageProvider
- localObjectKey
- thumbnailObjectKey
- durationSeconds
- width
- height
- isCover

但这是未来架构升级，不属于当前阶段。

## 7. 视频是否允许进入 Portfolio 首图

当前结论：不允许直接进入 Portfolio 首图。

未来最小允许策略：

- Portfolio 不直接播放视频；
- Portfolio 不自动加载视频 Blob；
- Portfolio 不 autoplay；
- Portfolio 不展示视频播放器；
- 如果某套房只有视频，可以显示轻量提示，例如“有本机视频”；
- 如果未来生成视频封面，也必须是本机生成或用户手动选择，不得自动从第三方视频链接抓取。

原因：

- Portfolio 是列表页，应保持轻量；
- 视频 Blob 加载会造成性能压力；
- 视频预览可能暴露更多室内隐私；
- 列表页播放视频会把“私人看房资料”推向“房源视频展示”。

## 8. 视频是否允许进入 AI

当前结论：不允许。

短期与中期均不做：

- AI 视频分析；
- 视频 OCR；
- 视频截图理解；
- 语音转文字；
- 房型识别；
- 装修判断；
- 采光判断；
- 人脸或声音识别。

原因：

- 视频可能包含人脸、声音、门牌号、合同、联系方式、房东或中介声音；
- 视频进入 AI 会显著提高隐私风险；
- HouseFolio 的 L3 定位是基于脱敏结构化数据做理解与表达，不是多模态房源审核；
- AI 不应被用于房源真实性认证、视频内容认证或装修质量判断。

未来如果做 AI 视频能力，必须另设独立授权、独立脱敏、独立风险评审和独立开关，不应继承照片模块默认行为。

## 9. 视频是否允许进入 JSON、ZIP、云同步

### JSON

不允许。

JSON 导入 / 导出只处理结构化 localStorage 数据。视频 Blob 不应进入 JSON。

### ZIP

当前不做。

未来如果做完整备份包，可以考虑：

- listings.json
- notes.json
- photos/
- videos/

但 ZIP 视频备份会显著增加文件体积和浏览器处理压力，应后置。

### 云同步

当前不做。

未来如果做，也必须满足：

- 默认关闭；
- 用户主动开启；
- 只走 lib/storage；
- 优先境内私有对象存储；
- 不公开；
- 不进入 Demo；
- 不进入 AI；
- 能单独删除；
- 能清晰显示云端同步状态；
- 能撤回同步授权。

## 10. 视频大小限制、播放策略、删除策略

未来最小实现建议如下。

### 文件类型

优先只支持：

- mp4
- webm

暂不支持：

- mov
- avi
- mkv
- m3u8
- 第三方视频链接

### 文件大小

建议 MVP 阶段单个视频限制为 30MB 或更低。

理由：

- IndexedDB 可以存 Blob，但大文件会明显增加浏览器存储压力；
- 个人作品不应提前承担重型视频管理责任；
- 过大的视频更适合用户留在本机相册或网盘，而不是进入 HouseFolio。

### 播放策略

- 只在 Detail 页播放；
- 用户点击后再加载；
- 不 autoplay；
- 不在 Portfolio 播放；
- 不在 Settings 播放；
- 不生成远程分享链接；
- 不上传云端转码；
- 不自动截图；
- 不自动提取音频。

### 删除策略

- 每个视频应支持单独删除；
- Settings 应支持清除全部本机媒体；
- 删除视频不能删除房源、笔记、评分、状态、通勤锚点、通勤结果；
- 删除结构化 JSON 数据不能自动删除视频 Blob，除非用户明确选择清除媒体资料。

## 11. 房东或中介视频的第三方内容边界

用户自己拍摄的视频，边界较清晰：

- 用户主动添加；
- 本机保存；
- 自己私用；
- 不公开；
- 不进入 AI；
- 不上传云端。

房东或中介发来的视频更谨慎：

允许用户作为私人看房资料本机保存，但必须限制：

- 不公开展示；
- 不二次传播；
- 不用于 PR；
- 不作为 Demo 素材；
- 不进入 AI；
- 不上传云端；
- 不做真实性背书；
- 不抓取视频链接；
- 不缓存第三方平台视频；
- 不生成公开分享页。

HouseFolio 可以是用户自己的私人资料夹，不能变成平台房源视频库。

## 12. 与 v2.0 本地优先路线的关系

视频问题应归入 v2.0 的“本地优先基础层”，不是 L1、L2 或 L3。

它服务于：

- 用户主动添加；
- 默认私有；
- 本地优先；
- 高敏资料本机持久化；
- 数据权利可见；
- 后续导出 / 导入路线。

它不应破坏：

- 不抓取；
- 不聚合；
- 不公开；
- 不撮合；
- 不认证；
- 不默认云同步；
- 不默认进入 AI；
- 不把 HouseFolio 做成云端房源相册。

## 14. 后续可能路线

如果未来决定实现视频，应按以下顺序推进：

1. Phase 3H-2：ListingMedia model review
2. Phase 3H-3：media storage provider migration plan
3. Phase 3H-4：video MVP implementation plan
4. Phase 3H-5：Detail local video panel minimal implementation
5. Phase 3H-6：Settings media summary update
6. Phase 3H-7：manual regression and closing log

但当前不建议马上进入这些阶段。

当前更稳妥的后续路线仍然是：

- Demo Mode 前置评审；
- 或 L2 comparison 前置评审；
- 或阶段 handoff 文档。

## 15. 最终结论

视频应该纳入 HouseFolio 长期“本机看房媒体资料”能力，但不应现在直接实现。

短期保持“看房照片｜本机保存”稳定。

中期如需实现视频，应合并为“看房照片/视频｜本机保存”，并严格限制为：

- 用户主动添加；
- 本机私有保存；
- 按 listingId 绑定；
- 只在 Detail 页按需播放；
- 不进 AI；
- 不云同步；
- 不公开；
- 不抓取；
- 不作为 Demo 素材；
- 不做真实性背书。

本阶段到此为止，只形成边界评审文档，不进入实现。
